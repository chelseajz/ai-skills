# Phase 5 深度分析规则

> 用于 Phase 5 深度分析，每个高评分事件必须按以下六个维度完整分析。执行时引用本文件。
> 同时引用 `reference/09-analysis-methodology.md`（咨询方法论）。

---

## 焦点事件筛选

```python
focus_events = [e for e in events if e.total_score >= 80]  # 焦点事件（必须深度分析）
watch_events = [e for e in events if 60 <= e.total_score < 80]  # 关注事件（简要分析）
```

## 分析前：框架选择 + 假设驱动

**对每个焦点事件，在六维度分析开始前**：

1. **选择分析框架**：从 `reference/09-analysis-methodology.md` 的框架矩阵中选择 1-2 个最相关框架
   - 格式：`[分析框架：VRIO + Porter's Five Forces]`
2. **提出初始假设**（1句话）
   - 格式：`[假设] X事件标志着Y趋势正在加速/放缓/转向`

---

## 论证结构：What → Why → So What

**每个维度的分析必须遵循 "What → Why → So What" 结构：**

| 层级 | 含义 | 示例 |
|------|------|------|
| **What** | 事实/数据：发生了什么？ | "Stripe 发布 MPP 开放协议" |
| **Why** | 原因/动机：为什么发生？ | "意图通过开放标准扩大开发者生态" |
| **So What** | 战略含义：对字节财经的启示？ | "字节若不自建开放协议，将依赖第三方标准，失去定价权" |

**质量检查**：如果只有 What 或 What + Why，缺少 So What，则不达标。

---

## 证据分级

每个关键主张必须标注证据等级（A-D），见 `reference/09-analysis-methodology.md` 第三节。
- ≥80 分事件的核心主张必须至少 B 级证据
- 缺失数据标注 "数据不可得"，禁止编造

## 六个维度深度分析框架（每个 ≥80 分事件必须全部完成）

### 维度 1: 信号（事实层）

**要求：**
- 用 1-2 句话客观描述事件核心事实
- 必须引用原文 quote（英文原文 + 中文翻译），标注发言人和出处
- 列出关键数据点（金额、时间、覆盖范围、增长率等）
- 区分"已确认事实"和"市场传闻/预期"
- **炒作/真实判断**：对该事件的真实影响力做快速判断，标注以下标签之一：
  → 【实质落地】— 产品已上线/牌照已发放/收购已完成
  → 【明确规划】— 公司已公开路线图、时间表、资源投入
  → 【概念阶段】— 仅为发布会发言、媒体通稿、概念演示
  → 【炒作信号】— 缺乏具体细节、无可验证里程碑、仅讲故事无数据
- 如果是【概念阶段】或【炒作信号】，必须额外说明"为什么可能只是炒作"（如无具体产品demo、无客户案例、无可验证KPI）

---

### 维度 2: 背景脉络 + 护城河分析（语境层）

**要求：**
- 事件发生的历史脉络：这个趋势/公司/产品之前走到哪一步了？
- 行业竞争格局：直接竞品同期做了什么？横向对比分析（参考 `knowledge-base/core-players.md` 和 `knowledge-base/competitors/` 中的竞品档案）
  → 例如分析 Mastercard 收购 BVNK 时，必须对比 Visa、Stripe、Adyen 同期在稳定币/加密支付领域的动作
  → 分析任何支付巨头动作时，必须明确列出 3-5 家直接竞品的同期对应动作
- **护城河变化分析（强制）**：竞品该动作是加深了哪条护城河？参考 `knowledge-base/methodology/moat-analysis.md` 中的护城河框架，按以下维度判断：
  → 网络效应（用户越多价值越高，如支付宝/微信支付的社交+支付飞轮）
  → 数据壁垒（交易数据→更好风控→更低定价→更大规模）
  → 牌照/合规壁垒（多区域牌照形成进入门槛）
  → 品牌心智（C端信任或B端开发者心智）
  → 规模经济（固定成本高、可变成本低）
  → 切换成本（B端集成一次用好，不会轻易换）
  → 判断该动作是"加深了哪条护城河"还是"用新维度开辟护城河"
  → 结论格式：`该动作主要加深了[护城河维度]，从[当前强度]变为[新强度]`
- 监管环境：哪些监管框架正在影响这个事件？
- 时间线：从事件首次出现到落地的完整时间线

---

### 维度 3: 影响评估（量化层 + 二阶影响）

**要求：**
- 对行业格局的影响：谁受益？谁受损？必须给出具体 TAM 或 TAM-at-risk 数字估算
- **二阶影响推导（强制）**：不能只分析一阶影响（"X受益"），必须继续推导：
  → 受益者下一步会做什么？（如：拿到牌照的机构会快速扩张还是谨慎试探？）
  → 受损者的反击动作是什么？（如：被挤压的支付公司会降价、差异化、还是退出？）
  → 二阶影响会波及哪些"无辜旁观者"？（如：稳定币监管收紧→影响DeFi→影响交易所→影响银行托管业务）
  → 至少推导 2 层因果链，格式：`一阶：X受益 → 二阶：X会做Y → 三阶：Z会被波及`
- 对上下游产业链的传导效应：上游供应商、下游客户、周边生态
- 量化评估：如涉及金额/用户/市场份额，必须给出具体数字或合理估算
  → **禁止只说"影响较大/有限"，必须给出至少一个量化锚点**（如：涉及金额、用户规模、市场份额%、增长率等）
- **产品级功能/特性追踪（强制）**：对于竞品发布的具体产品功能（如支付宝AI Skill、微信"碰一下"支付场景拓展、Meta AI购物模式、Stripe MPP协议等），必须分析：
  → 该功能解决了用户的什么具体痛点？现有方案有什么不足？
  → 该功能的技术门槛和复制难度如何？
  → 该产品功能在行业中的领先程度（首创/跟随/追赶）？
- **对字节财经的影响（必须落实到具体产品名，强制）**：
  → 明确到字节财经具体产品：抖音支付、放心借（消费信贷）、月付/抖音月付（BNPL）、TikTok Shop支付（东南亚/英国）、抖音保险平台（保险经纪）、香港SVF数字钱包
  → **每个受影响的字节产品必须分析以下 4 项**：
    1. **竞争压力**：该竞品动作对该产品的直接威胁是什么？威胁等级（高/中/低）
    2. **可借鉴点**：该功能/模式是否可以复用到字节产品？复制难度和预期收益
    3. **字节优势**：字节在该场景有什么独特优势（流量、数据、AI、场景嵌入）？
    4. **行动建议**：短期（1个月内）和中期（3个月内）应该做什么？

示例分析微信"碰一下"：
> - 对抖音支付：威胁等级中（线下场景挤压），但抖音支付主要在电商闭环内，线下不是当前核心
> - 可借鉴点："碰一下"的NFC/蓝牙近场支付技术方案可直接参考，字节已有支付牌照
> - 字节优势：抖音电商用户已习惯站内支付，若拓展到店/本地生活场景可复用
> - 行动建议：短期调研"碰一下"技术方案和用户数据；中期评估抖音本地生活是否需要近场支付能力

### 维度 4: 趋势预判（前瞻层）

**要求：**
- 必须给出明确判断方向：加速 / 放缓 / 转向 / 伪趋势（不能只说"在发展"/"值得关注"）
- 未来 3-6 个月具体预测（给出可验证的事件和时间点）
- 关键变量：哪些因素可能加速或延缓这个趋势？
- 类比历史案例：是否有历史上类似的事件？最终结果如何？
- **每条预测必须标注信心等级**：高（>70%）/ 中（40-70%）/ 低（<40%）+ 验证时间点（YYYY-MM）
- **证伪条件必须基于最新事实**：写证伪条件前，必须先搜索该趋势最新进展，确认已有证伪条件是否已触发
  → 例如 Agentic Commerce 趋势，Meta 已发布 AI 购物模式（2026年3月），不能再以"Meta/Instagram未上线AI购物"作为证伪条件
  → 必须基于当月最新的事实调整预测和证伪条件

---

### 维度 5: 缺口与机会 + 竞争窗口期（洞察层）

**要求：**
- 市场普遍忽略了什么？（主流媒体报道的盲区）
- 竞争格局中的空白点：哪些玩家还没表态？哪些市场还无人覆盖？
- 规则/技术变化带来的新机会或新壁垒
- **竞争窗口期分析（强制）**：
  → 判断当前是"进入窗口"还是"退出窗口"：
    - **进入窗口**：新玩家还有机会进入、标准尚未收敛、先发者优势未稳固
    - **退出窗口**：格局已定、壁垒已固化、后进入者代价高昂
  → 给出窗口期剩余时间估算（如："还有 6-12 个月窗口期"、"窗口正在关闭"）
  → 什么信号出现意味着窗口彻底关闭？
- **必须给出至少 1 条非共识观点**，格式要求：
  1. 先写**市场共识：**...（主流观点是什么）
  2. 再写**我的判断：**...（为什么不同）
  3. 再写**验证信号：**...（什么数据/事件会证明我对或错）
  → 不能写"值得关注""可能重要"等模糊表述，必须有明确方向
  → **非共识观点的质量自检**：如果去掉这个观点换成"值得关注"也能读得通，说明这不是真正的非共识，必须重写

---

### 维度 6: 行动建议（落地层）

**要求：**
- 必须具体到可执行动作（如："调研 XX 公司的 API 文档"、"访谈 XX 行业从业者"、"POC 验证 XX 场景"）
- 短期建议（1 个月内可以做的调研/访谈/POC）
- 中期建议（3 个月内的战略决策节点）
- 风险预警：什么信号出现时需要重新评估判断？
- **每条建议必须包含**：具体调研对象 + 时间窗口 + 成功/失败指标
- **建议优先级矩阵**：按"影响力×可执行性"排序，标注每个建议的优先级（P0必须做/P1建议做/P2可选）
- **建议必须关联到具体字节产品**：每条建议明确指向抖音支付/放心借/月付/TikTok Shop支付/抖音保险/香港SVF中的哪个产品

## 跨事件关联分析（焦点事件之间的交叉分析）

对焦点事件进行聚类分析，识别以下类型的关联：
- 竞品联动：同一赛道多家公司的同期动作（如多个 AI 支付产品同时发布）
- 监管共振：多个地区的监管框架同时推进（如全球稳定币监管收敛）
- 技术-应用链条：上游技术突破 + 下游商业化落地同时发生
- 并购-整合链：收购事件 + 被收购方后续动作的连续性
- 政策-市场互动：监管政策发布后市场参与者的应对动作
- **零和博弈**：一方增长直接导致另一方受损（如微信支付拓展线下场景→支付宝份额下降）

**输出要求：**
1. 对每条关联输出：→ [关联类型] 事件A 与 事件B 的关联分析（2-3 句话，不能只列事实，必须给出推论）
2. **至少做 1 条"因果链推导"**：
   事件A（因）→ 事件B（果/放大器）→ 对行业格局的连锁影响
   例如：`Stripe MPP 开放协议（事件A） → 更多中小支付公司采用标准接口（事件B） → 头部支付公司壁垒从'技术独家'转向'规模和合规'，Stripe 凭借开发者生态和交易规模优势反而加深护城河`
3. 关联分析必须指向至少 1 条趋势判断，不能孤立存在

## 趋势判断与非共识观察

基于全部焦点事件 + 跨事件关联分析：
- **识别 3-5 条核心趋势**，每条趋势必须：
  - 有明确的判断方向（"X 将加速/放缓/转向"，不能只说"X 在发展"）
  - 有至少 2 个事件作为论据支撑
  - 给出可验证的时间窗口（"预计 Q3/Q4 2026 会发生..."）
  - 标注趋势阶段：萌芽期 / 加速期 / 成熟期 / 衰退期
  - **竞争格局重塑分析（强制）**：该趋势会如何改变竞争格局？
    → 谁将成为赢家？谁会被淘汰？（必须点名 2-3 家具体公司）
    → 新进入者的门槛是什么？门槛在升高还是降低？
    → 该趋势对字节财经的战略启示：应该进入/加深投入/观望/退出？
  - **趋势间相互作用**：识别 2-3 条趋势之间的相互影响关系
    → 趋势A 加速 → 趋势B 被加速/被抑制/无关？为什么？
    → 例如：`稳定币监管收敛（趋势A）加速 → Agentic Payment 采用稳定币结算（趋势B）加速，因为合规路径清晰后AI支付代理更愿意使用稳定币`

- **生成 3-5 条非共识观察**，每条必须按以下格式：
  1. **市场共识**：主流观点/媒体报道的叙事是什么（1句话）
  2. **我的判断**：为什么不同（1-2句话，必须有明确方向）
  3. **支撑论据**：至少 2 个数据/事件/逻辑链
  4. **证伪条件**：什么信号出现时证明我错了（具体事件+时间窗口）
  5. **对字节启示**：具体到哪个业务线应该关注/行动，以及行动建议
  → 禁止写"值得关注""有潜力""应持续跟踪"等空泛表述
  → **非共识观点必须满足以下至少一项**：
     a) 指出被市场忽略的关键变量
     b) 给出与主流预测相反的方向判断
     c) 指出看似利好实则有隐患的信号
     d) 指出看似利空实则是机会的信号

## 假设验证 + 红队审查（分析末尾必须包含）

### 假设验证

对分析开始时提出的每个初始假设，给出验证结论：
```markdown
## 假设验证
- **[假设]** X事件标志着Y趋势正在加速
  - **[支持证据]** 事实A + 事实B
  - **[反证]** 但Z事实构成挑战：...
  - **[结论]** 假设被{证实/部分证实/推翻}。理由：...
```

### 红队审查

执行以下 5 项审查：
1. **最强反驳**：如果要从该分析中找出 1 个最薄弱的论据，是什么？
2. **遗漏检查**：有哪些关键玩家/事件/数据在本月未被覆盖，可能改变结论？
3. **方向性错误风险**：分析结论的方向（加速/放缓/转向）有无反向可能？
4. **字节偏差检查**：是否因为字节是服务对象而高估了某些影响或低估了某些威胁？
5. **数据时效性**：分析中引用的数据是否过时？

```markdown
## 红队审查
- **最薄弱论据**：...
- **潜在遗漏**：...
- **方向性风险**：...
- **字节偏差**：...
- **数据时效性**：...
```
