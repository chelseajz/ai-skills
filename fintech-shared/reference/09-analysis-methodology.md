# 咨询级分析方法论

> 用于 Phase 5 深度分析中的论证质量提升。执行时引用本文件。

---

## 一、分析框架选择矩阵

在开始六维度分析前，先为每个焦点事件选择 1-2 个最相关的分析框架，显式标注在分析开头。

| 分析场景 | 选用框架 | 说明 |
|----------|----------|------|
| 竞品公司战略动作 | **SWOT** + **Value Chain** | 评估该动作的优劣势及在价值链中的位置 |
| 行业进入/壁垒 | **Porter's Five Forces** | 评估五力变化，判断进入门槛升降 |
| 护城河/核心竞争力 | **VRIO** | 价值-稀缺-不可模仿-组织，判断竞争优势可持续性 |
| 监管政策变化 | **PESTEL**（聚焦P+L） | 政治/法律环境变化 + 对Economic/Tech的传导 |
| 新市场/新赛道 | **TAM-SAM-SOM** | 量化市场空间，判断是否值得进入 |
| 技术成熟度评估 | **Gartner Hype Cycle** | 判断技术处于触发期/膨胀期/低谷期/成熟期 |
| 并购交易分析 | **Value Chain** + **Strategic Group** | 并购在价值链中的整合逻辑 + 竞争格局重塑 |
| 产品/功能发布 | **JTBD** + **Benchmarking** | 该功能解决了用户什么任务 + 与竞品功能KPI对比 |

**选择原则**：
- 每个焦点事件至少标注 1 个框架，最多 3 个
- 框架之间必须互补（不重叠）
- 在分析开头显式标注：`[分析框架：VRIO + Porter's Five Forces]`

## 二、"What → Why → So What" 论证结构

**每个维度的分析必须遵循此结构，不得跳过。**

| 层级 | 含义 | 示例 |
|------|------|------|
| **What** | 事实/数据：发生了什么？ | "Stripe 发布 MPP 开放协议，允许中小支付公司接入其清结算网络" |
| **Why** | 原因/动机：为什么发生？ | "Stripe 意图通过开放标准降低中小支付公司的集成门槛，扩大其开发者生态，形成网络效应壁垒" |
| **So What** | 含义/影响：这意味着什么？对读者（字节财经）的启示？ | "这意味着支付行业的竞争门槛从'技术独家'转向'生态规模'——字节若不自建开放协议，将依赖第三方标准，失去定价权。建议在6个月内评估是否加入MPP或自建替代方案" |

**质量检查**：如果分析只有 What（事实描述）或 What + Why（事实+原因），缺少 So What（战略含义），则不达标，必须重写。

**So What 链深度要求**（8条判断规则，按顺序执行）：

1. **数据 → 行为变化**：这个数字说明了用户/企业行为的什么变化？
2. **行为 → 动机推断**：行为变化背后的动机是什么？（成本/风险/增长/合规？）
3. **动机 → 竞争格局**：这个动机会如何改变竞争者之间的关系？
4. **格局 → 受益/受损者**：谁在这个新格局中获益？谁受损？
5. **受益者 → 下一步动作**：受益者接下来最可能做什么？
6. **受损者 → 反击策略**：受损者会如何反击？
7. **二阶传导 → 无辜旁观者**：哪些"旁观者"会被波及？
8. **最终 → 对字节启示**：字节财经应该做什么具体动作？

## 三、证据分级标准

每个关键主张必须标注证据等级：

| 等级 | 名称 | 定义 | 示例 |
|------|------|------|------|
| **A** | 一手确认 | 来自公司官方声明、财报原文、监管官网原文 | "Stripe 官方新闻稿确认 MPP 协议发布" |
| **B** | 二手可靠 | 来自权威媒体（Reuters/Bloomberg/FT）的独立报道 | "Reuters 报道 Stripe MPP 协议细节，引用公司高管发言" |
| **C** | 二手一般 | 来自行业媒体、博客、社交媒体的报道 | "TechCrunch 分析文章解读 Stripe MPP 影响" |
| **D** | 未验证 | 匿名信源、社交媒体传闻、无明确来源的分析 | "有消息称 Stripe 正在测试新功能" |

**规则**：
- ≥80分事件的核心主张（信号、影响金额、市场份额）必须至少 B 级证据
- 非共识观点可以使用 C 级证据，但必须有至少 2 个独立来源交叉验证
- D 级证据**不得**作为关键论据，只能作为补充提及
- 缺失数据必须明确标注 "数据不可得"，**禁止**编造或估算后不标注

## 四、假设驱动分析方法

**核心思路**：先形成初步假设，再用数据验证或推翻。

**步骤**：

1. **提出初始假设**（分析开始前，1句话）
   - 格式：`[假设] X事件标志着Y趋势正在加速/放缓/转向`
   - 例如：`[假设] Stripe MPP 协议发布标志着支付清结算行业正在从封闭生态转向开放标准`

2. **寻找支持证据**（分析中列出）
   - 列出至少 2 条支持假设的事实/数据

3. **寻找反驳证据**（分析中必须包含）
   - 主动搜索可能推翻假设的证据
   - 格式：`[反证] 但Z事实对该假设构成挑战：...`

4. **给出验证结论**（分析末尾）
   - 格式：`[结论] 假设被{证实/部分证实/推翻}。理由：...`
   - 如果部分证实，说明哪些条件成立、哪些不成立

**不要求每个事件都做完整的假设驱动，但对以下类型的事件必须执行**：
- 行业级趋势判断（如"AI支付代理正在兴起"）
- 市场格局变化的判断（如"稳定币正在成为主流支付工具"）
- 非共识观点

## 五、叙事质量规则

### 5.1 标题写结论，不写标签

```
❌ 差：Stablecoin 概述
✅ 好：稳定币月度交易量创历史新高，亚太支付 adoption 是核心驱动力

❌ 差：监管动态
✅ 好：香港虚拟资产牌照框架正式生效，首批5家机构获批

❌ 差：竞品分析
✅ 好：Stripe MPP 协议推动支付清结算从封闭生态转向开放标准
```

### 5.2 金字塔原理（结论先行）

每个分析段落的第一句必须是结论，后续句子是支撑论据。

```markdown
**结论句**：该事件加深了网络效应护城河，从"区域覆盖"维度构建了进入壁垒。

**论据1**：接入 MPP 的支付公司数量从 3 月 12 家增至 4 月 28 家，月增速 133%。

**论据2**：每新增一家支付公司，网络对现有参与者的价值提升约 2.3%（基于交易量交叉引用数据）。

**论据3**：新进入者需要同时达到至少 50 家商户+20 家支付公司的双边规模才能形成可比网络。
```

### 5.3 数据 → 心理 → 战略链

每个洞察必须连接：数据 → 行为/心理 → 战略含义。

```
❌ 差：移动支付交易额增长 20%，建议关注移动支付赛道。

✅ 好：移动支付交易额同比增长 20%，但线下扫码支付增速从 35% 降至 8%，**这表明**
用户正从线下场景转向线上/近场支付。**因此**，字节若仅在电商闭环内优化支付体验，
将错失线下场景增量——建议在本地生活场景中优先布局"碰一下"类近场支付能力。
```

## 六、反幻觉协议

### 6.1 零容忍规则

- 报告中所有数据、事件、URL 必须来自 Phase 1-3 搜索结果或 Phase 2 咨询报告
- **禁止**编造、估算、模拟数据。缺失数据标注 "数据不可得"
- 每项陈述必须可追溯回原始数据源（被要求时能指出来源 URL）
- 区分"已确认事实"和"推论/预测"，两者用不同句型

### 6.2 可追溯性要求

每个焦点事件的六维度分析末尾，必须标注关键数据的来源：

```markdown
**数据来源**：
- 交易金额：[Stripe 官方新闻稿](URL)
- 市场份额估算：[Reuters 报道](URL) + [Statista 数据](URL)
- 护城河判断：基于 VRIO 框架分析（分析者判断，非外部来源）
```

### 6.3 因果关系标注

使用 SVO（主谓宾）句型描述因果关系，明确区分相关性和因果性：

```
✅ 因果：X 事件发生后，Y 公司在两周内宣布了应对方案（有时间先后+逻辑链）。
⚠️ 相关：X 事件和 Y 事件发生在同月，两者都涉及支付行业（仅时间巧合）。
```

## 七、红队审查（Adversarial Review）

在分析完成、报告生成前，执行以下红队审查：

1. **最强反驳**：如果要从该分析中找出 1 个最薄弱的论据，是什么？
2. **遗漏检查**：有哪些关键玩家/事件/数据在本月未被覆盖，可能改变结论？
3. **方向性错误风险**：分析结论的方向（加速/放缓/转向）有无反向可能？
4. **字节偏差检查**：是否因为字节是服务对象而高估了某些影响或低估了某些威胁？
5. **数据时效性**：分析中引用的数据是否过时？是否有当月更新的数据未纳入？

**输出格式**（在 phase5-analysis.md 末尾追加）：

```markdown
## 红队审查
- **最薄弱论据**：[描述]
- **潜在遗漏**：[描述]
- **方向性风险**：[描述]
- **字节偏差**：[描述]
- **数据时效性**：[描述]
```
