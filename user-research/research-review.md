# Skill: research-review

## 描述
调研报告审核。作为独立审核者，评估研究报告的证据链完整性、逻辑严谨性、洞察深度、信息密度和业务价值，输出结构化审核意见。

## 调用方式
```
/research-review --report_content="报告内容" --original_data="原始数据" --review_focus="审核重点" --business_context="业务背景"
```

## 参数说明
- **report_content**: 报告内容（全文或指定章节）
- **original_data**: 原始数据/编码表（可选，用于验证）
- **review_focus**: 审阅重点（可选，如全面审阅/仅审结论/仅审逻辑）
- **business_context**: 业务背景（用于评估业务价值）

## 输出内容
1. 审阅概览（八维度评分+总体评分+发布建议）
2. 详细审阅意见（P0/P1/P2 问题+亮点）
3. 红队挑战专项（每个核心结论的最强反驳）
4. Claim 分类验证（Fact/Finding/Insight/Hypothesis/Recommendation）
5. 信息密度审计
6. 修改优先级矩阵
7. 复审检查清单

## 审阅维度

| 维度 | 说明 |
|------|------|
| 证据链完整性 | 每个结论有≥2个独立数据源、定量有统计证据、定性有语录 |
| 逻辑严谨性 | 推理无跳跃、相关与因果区分、替代解释被考虑 |
| 洞察深度 | So-What链≥3层、Observation/Insight/Implication明确区分 |
| 信息密度 | 无空话套话、每段有数据点或语录、删除测试通过 |
| 业务价值 | 有可行动建议、有ICE评分、有成功指标 |
| 可信度与透明度 | 置信度标注准确、局限性明确、样本偏差披露 |
| 传播质量 | 简洁可扫描、保留用户声音、不扁平化为抽象标签 |
| 交付系统质量 | 主报告独立成立、图表位置恰当、命名基于纳入逻辑 |

## Claim 分类标准

每个重要陈述必须归类为：

- **Fact**: 直接由观测数据支持
- **Finding**: 有边界模式，由证据支持
- **Insight**: 解释"为什么"
- **Hypothesis**: 前瞻性或弱支持解读
- **Recommendation**: 基于发现提出的行动

**注意**: 不要将 Hypothesis 标记为 Finding。

## 置信度标准

- **High**: 证据强且一致，主要局限不逆转结论，多源交叉验证
- **Medium**: 模式可信但暴露于样本/测量/解读限制，证据部分间接
- **Low**: 证据稀疏/冲突/高偏，假设生成有用

## 发布标准

- [ ] 无 P0 严重问题
- [ ] P1 中等问题 ≤3 个
- [ ] 总体评分 ≥5/8
- [ ] 核心结论有充分证据
- [ ] 红队挑战被充分回应
- [ ] Claim 分类正确

## 严重问题清单（P0）

- 核心数字无法追溯
- 核心数字使用不一致或未命名的分母
- 局限性被省略或轻描淡写
- 关键建议未基于发现
- 定性证据被当作人口代表性使用
- 面向读者的交付物仍包含明显的格式残留或图表放置错误
